# Butler (дворецкий) — Product Requirements + Architecture Specification

## 1. Статус документа

**Тип документа:** PRD + Architecture (продукт + архитектура)  
**Версия:** 0.1  
**Статус:** Draft / Baseline  
**Назначение:** зафиксировать целевую модель системы Butler для первой полноценной архитектурной итерации.

---

## 2. Резюме

Butler (дворецкий) — это личный долгоживущий агент, ориентированный на self-hosted (самостоятельно размещаемую) эксплуатацию, работающий через Telegram и Web UI (веб-интерфейс), использующий WebSocket-first transport (транспорт с приоритетом WebSocket) для взаимодействия с моделями, а также собственную подсистему памяти и набор инструментов.

Ключевая цель Butler — быть не просто чат-ботом, а постоянным персональным агентом с состоянием, способным:
- вести длительный контекст взаимодействия;
- использовать tools (инструменты) для выполнения задач;
- помогать пользователю настраивать и поддерживать саму систему Butler;
- работать как с OpenAI, так и с локальными моделями;
- оставаться управляемой, наблюдаемой и безопасной системой, а не набором ad hoc (разрозненных) скриптов.

В первой версии Butler фокусируется на личном использовании, двух каналах взаимодействия и ограниченном, но качественно реализованном наборе инструментов: browser automation (автоматизация браузера), HTTP / web, self-inspection / doctor (самодиагностика и помощь в настройке самой системы), а также на подсистеме управления учётными данными и отложенного разрешения секретов для безопасной авторизации без раскрытия секретов агенту.

---

## 3. Цели продукта

### 3.1 Основные цели

1. Построить личного долгоживущего агента с устойчивой памятью и длительным контекстом.
2. Сделать систему self-hosted по умолчанию и docker-oriented (ориентированной на Docker) на всех уровнях.
3. Использовать WebSocket-first transport для уменьшения накладных расходов, снижения latency (задержки) и улучшения tool-heavy flows (сценариев с большим числом вызовов инструментов).
4. Обеспечить поддержку облачных провайдеров OpenAI API, OpenAI Codex / ChatGPT OAuth, GitHub Copilot, а также локальных моделей.
5. Построить архитектуру, пригодную для дальнейшего роста: новые инструменты, новые режимы автономности, новые каналы.

### 3.2 Дополнительные цели

1. Дать агенту возможность помогать пользователю диагностировать и настраивать Butler.
2. Создать архитектуру, где память, инструменты и транспорт не смешаны в один монолитный процесс.
3. Сохранить возможность дальнейшего перехода к более сложной оркестрации и расширенной автоматизации без полной переработки ядра системы.
4. Обеспечить безопасное использование паролей, токенов, cookies и storage state без помещения секретов в модельный контекст.

---

## 4. Не-цели первой версии

В первую версию Butler **не входят**:
- мультиагентная платформа как основной сценарий;
- корпоративный multi-tenant (многопользовательский многотенантный) режим;
- автономное саморасширение набора tools;
- широкий набор интеграций вроде calendar / email / database;
- сложные распределённые облачные сценарии.

Эти направления рассматриваются как future scope (будущая область развития), но не входят в baseline V1.

## 5. Пользовательский сценарий

### 5.1 Основной пользователь

Один владелец self-hosted инстанса Butler, использующий систему как личного агента.

### 5.2 Основные способы взаимодействия

- Telegram — основной асинхронный канал сообщений.
- Web UI — основной интерактивный интерфейс для настройки, наблюдаемости и управления состоянием агента.

### 5.3 Типовые сценарии

1. Пользователь пишет агенту задачу в Telegram.
2. Агент продолжает существующую сессию, поднимает релевантную память, вызывает web / browser tools и возвращает результат.
3. Пользователь открывает Web UI для просмотра истории, памяти, диагностических отчётов и статуса системы.
4. Пользователь просит агента проверить его конфигурацию Butler.
5. Агент использует self-inspection / doctor tool для анализа собственного состояния, конфигурации и зависимостей, после чего предлагает действия по исправлению.

## 6. Архитектурные принципы

1. **WebSocket-first, but not state-owned by transport**  
   WebSocket является предпочтительным каналом общения с моделью, но транспорт не считается источником истины для памяти, сессий или состояния системы.

2. **Memory is first-class infrastructure**  
   Память — это отдельный архитектурный слой, а не просто история сообщений.

3. **Tools are controlled execution surfaces**  
   Инструменты считаются контролируемыми исполнительными поверхностями, а не просто функциями.

4. **Docker-oriented deployment**  
   Все основные части системы разворачиваются как Docker containers (докер-контейнеры).

5. **Separation of concerns (разделение ответственности)**  
   Каналы, оркестрация, модельный транспорт, память и инструменты разделены на отдельные подсистемы.

6. **Self-hosted first**  
   Все архитектурные решения должны работать в модели полностью self-hosted развертывания.

7. **Progressive autonomy (постепенная автономность)**  
   Система должна поддерживать разные режимы автономности, а не только один жёстко заданный режим.

## 7. Высокоуровневая архитектура

### 7.1 Верхнеуровневые компоненты

1. **Channel Adapters (адаптеры каналов)**
   - Telegram Adapter
   - Web UI Backend Adapter

2. **Ingress / API Layer**
   - принимает входящие события;
   - нормализует формат сообщений;
   - передаёт их в session/orchestration layer.

3. **Session Service**
   - ведёт session identity (идентификацию сессии);
   - хранит состояние активных запусков;
   - обеспечивает порядок выполнения и блокировки.

4. **Orchestrator**
   - управляет жизненным циклом run (запуска);
   - собирает контекст;
   - выбирает модель и набор инструментов;
   - принимает решения о вызовах tools и памяти.

5. **Model Transport Layer**
   - реализует WebSocket-first взаимодействие с OpenAI API там, где это поддерживается;
   - поддерживает OpenAI Codex / ChatGPT OAuth и GitHub Copilot как отдельные cloud providers;
   - поддерживает локальные модели;
   - скрывает различия между провайдерами.

6. **Tool Broker**
   - валидирует tool calls;
   - маршрутизирует их к tool runners;
   - логирует вызовы;
   - ограничивает доступ и ресурсы.

7. **Credential Broker + Secret Store**
   - хранят metadata и secret material раздельно;
   - разрешают credential aliases в runtime secrets только на системном слое;
   - проверяют approval policy, allowed domains и allowed tools;
   - аудируют каждое использование alias.

8. **Memory Service**
   - управляет Transcript Store, Working Memory, Episodic Memory и Profile Memory.

9. **Restart Helper**
   - отдельный узкий control-plane service;
   - имеет доступ к Docker socket только для allowlisted restart-операций;
   - не хранит память, не исполняет tools и не принимает orchestration decisions;
   - используется Web UI settings flow через orchestrator для перезапуска изменённых сервисов без передачи широкого Docker control в orchestrator.

10. **Observability + Diagnostics**
   - системные логи;
   - метрики;
   - трассировка запусков;
   - self-inspection / doctor.

## 8. Логическая схема потока

### 8.1 Поток входящего пользовательского запроса

1. Сообщение поступает из Telegram или Web UI.
2. Channel Adapter нормализует событие.
3. Если пользователь явно передал credential alias, ingress layer выделяет отдельный credential context, не смешанный с обычным текстом запроса.
4. Session Service определяет session key и получает lock / lease (блокировку / аренду обработки).
5. Orchestrator запрашивает краткую session summary, working memory и релевантные episodic/profile memories.
6. Orchestrator формирует context bundle (пакет контекста) и безопасный credential capability view, не содержащий секретных значений.
7. Model Transport Layer отправляет запрос модели через WebSocket-first transport.
8. Если модель инициирует tool call, Tool Broker валидирует и исполняет инструмент.
9. Если tool args содержат deferred credential references, Tool Broker и Credential Broker проверяют policy, domain и autonomy mode, после чего разрешают ссылки только внутри runtime.
10. Tool result возвращается в orchestrator и добавляется как новый input item (элемент ввода) в продолжающуюся модельную сессию.
11. Модель завершает ответ.
12. Transcript Store получает полную запись хода.
13. Memory pipeline извлекает и сохраняет новые memory items.
14. Credential audit log получает запись об использовании alias.
15. Ответ возвращается пользователю.

## 9. Модель каналов

### 9.1 Telegram

Telegram является главным пользовательским каналом V1. Требования:
- устойчивое сопоставление пользователя и сессии;
- поддержка долгих разговоров;
- возможность отправки как кратких, так и развёрнутых ответов;
- безопасное подтверждение действий для опасных операций.

#### 9.1.1 Подтверждение действий (Approval UX)

Когда агент запрашивает подтверждение пользователя (credential usage, mutating tool call, опасная операция), Butler отправляет Telegram-сообщение с inline keyboard buttons.

**Минимальный набор кнопок:**
- **Approve** — подтвердить выполнение;
- **Reject** — отклонить действие;
- **Details** — показать расширенное описание того, что агент собирается сделать.

**Правила:**
- Сообщение с запросом подтверждения содержит краткое описание действия: tool name, target domain (если применимо), credential alias (если применимо), risk level.
- Кнопки привязаны к конкретному `tool_call_id` через callback data; повторное использование кнопки невозможно.
- **Timeout:** если пользователь не ответил в течение configurable timeout (рекомендуемое значение по умолчанию — 5 минут), действие автоматически отклоняется. Агент получает результат `policy_denied` с причиной `approval_timeout`.
- **Cancel:** пользователь может отменить ожидающее подтверждение текстовой командой `/cancel` или кнопкой Reject.
- **Audit:** каждый запрос подтверждения и его результат (approved / rejected / timed_out) фиксируются в audit log с timestamp, tool_call_id, alias (если есть) и решением пользователя.
- **Одновременные запросы:** если несколько действий ожидают подтверждения, каждое отображается как отдельное сообщение с собственным набором кнопок. Порядок подтверждения не фиксирован.

### 9.2 Web UI

Web UI требуется для:
- просмотра истории сессий;
- просмотра памяти;
- мониторинга tools;
- показа диагностических отчётов Butler;
- управления конфигурацией и режимами автономности;
- планирования и запуска helper-mediated restart для сервисов, которым нужен restart после settings changes.

Web UI не должен быть просто “ещё одним чатом”; это административно-пользовательский интерфейс системы.

## 10. Модель провайдеров и транспорта

### 10.1 Поддерживаемые провайдеры

V1 должна поддерживать:
- OpenAI API;
- OpenAI Codex / ChatGPT OAuth;
- GitHub Copilot;
- локальные модели.

### 10.2 WebSocket-first transport

WebSocket-first transport является обязательным архитектурным требованием для общения с облачными моделями, где это поддерживается.

#### Цели WebSocket-first transport

1. Снижение latency.
2. Эффективное продолжение длинных tool-heavy workflows.
3. Уменьшение необходимости повторно пересылать весь контекст.
4. Более естественная поддержка потоковых ответов.

### 10.3 Ограничение

Даже при использовании WebSocket-first transport:
- session truth (истина о сессии) хранится внутри Butler;
- memory truth хранится внутри Butler;
- модельный transport layer не является единственным источником состояния.

### 10.4 Абстракция транспорта

Model Transport Layer должен предоставлять единый интерфейс:
- `start_run()`
- `continue_run()`
- `submit_tool_result()`
- `stream_events()`
- `cancel_run()`

Это необходимо для одинаковой работы с OpenAI API, OpenAI Codex, GitHub Copilot и локальными провайдерами.

## 11. Модель памяти

Butler использует многослойную memory architecture (архитектуру памяти), где разные классы памяти выполняют разные функции.

### 11.1 Transcript Store

**Назначение:** полная история взаимодействия.

**Содержит:**
- пользовательские сообщения;
- ответы агента;
- tool calls;
- tool outputs;
- системные события;
- диагностические записи.

**Роль:** источник истины для истории выполнения.

**Особенности:**
- append-only или максимально близкая к этому модель;
- полная трассируемость;
- пригодность для аудита и последующего анализа.

### 11.2 Working Memory

**Назначение:** краткоживущий контекст текущей задачи.

**Содержит:**
- текущую цель;
- активные сущности;
- промежуточные шаги;
- незавершённые действия;
- технический scratch state (рабочий черновой контекст).

**Роль:** помогает агенту удерживать текущий task state (состояние задачи) без раздувания долгосрочной памяти.

### 11.3 Episodic Memory

**Назначение:** хранение значимых завершённых эпизодов.

**Содержит:**
- краткие записи о том, что было сделано;
- важные ошибки и успешные паттерны;
- результаты прошлых действий;
- события, полезные для будущих похожих задач.

**Роль:** recall by similarity and relevance (воспроизведение по смысловой близости и релевантности).

### 11.4 Profile Memory

**Назначение:** долговременные устойчивые сведения о пользователе и системе.

**Содержит:**
- пользовательские предпочтения;
- конфигурационные договорённости;
- устойчивые свойства окружения;
- знания о конкретной установке Butler.

**Роль:** стабильный персональный и системный профиль.

## 12. RAG и векторный поиск

### 12.1 Принцип

RAG (retrieval-augmented generation — генерация с извлечением контекста) в Butler рассматривается как retrieval mechanism (механизм извлечения), а не как полная замена памяти.

### 12.2 Базовый подход

Используется гибридная схема:
- structured storage (структурированное хранение) — источник истины;
- vector retrieval (векторное извлечение) — поиск по смысловой близости;
- metadata filtering (фильтрация по метаданным);
- при необходимости keyword/full-text search (поиск по ключевым словам / полнотекстовый поиск).

### 12.3 Выбранная технология

Так как в системе уже предполагается использование PostgreSQL, для векторного слоя рекомендуется `pgvector`.

### 12.4 Что индексируется

В векторный поиск рекомендуется отдавать:
- episodic memories;
- summaries длинных диалогов;
- knowledge/document chunks;
- релевантные длинные tool outputs.

Не рекомендуется без разбора индексировать весь raw transcript.

### 12.5 Вывод

RAG в Butler **нужен**, но только как часть memory pipeline, а не как единственная модель памяти.

## 13. Хранилища данных

### 13.1 PostgreSQL

Используется как primary system of record (основное системное хранилище).

Предварительные сущности:
- sessions
- runs
- messages
- tool_calls
- tool_outputs
- memory_working
- memory_episodes
- memory_profile
- memory_chunks
- memory_links
- diagnostics_reports
- credential_records
- credential_audit_logs

### 13.2 pgvector

Используется как слой векторного поиска поверх PostgreSQL.

### 13.3 Redis или аналог

Используется как слой быстрого состояния:
- locks / leases;
- временные execution states;
- short-lived caches;
- rate limits.

### 13.4 Secret Store

Используется как отдельное хранилище secret material:
- username / password pairs;
- API tokens;
- cookie bundles;
- browser storage state;
- OAuth tokens.

Secret Store не должен раскрывать секреты orchestrator, model transport или LLM-контексту; фактические значения выдаются только runtime инструмента через Credential Broker.

## 14. Модель инструментов

### 14.1 Инструменты V1

Первая версия Butler должна включать:
1. **Browser Automation Tooling**
2. **HTTP / Web Tooling**
3. **Self-Inspection / Doctor Tooling**

### 14.2 Browser Automation

Назначение:
- открытие страниц;
- чтение DOM / текста;
- выполнение навигации;
- при необходимости подготовка к будущей browser-based automation.

### 14.3 HTTP / Web

Назначение:
- чтение публичных и разрешённых HTTP endpoints;
- интеграция с внешними веб-сервисами;
- получение JSON / HTML / текстовых данных.

### 14.4 Self-Inspection / Doctor

Назначение:
- анализ состояния самой системы Butler;
- проверка доступности сервисов и контейнеров через safe diagnostics surfaces;
- диагностика конфигурации;
- объяснение пользователю, как исправить обнаруженные проблемы.

Границы:
- doctor не должен выполнять Docker restart/stop операции;
- doctor не должен владеть Docker socket;
- любые container lifecycle actions должны идти через отдельный узкий control-plane helper.

Этот инструмент является принципиально важным дифференциатором системы.

### 14.5 Credential-aware tool execution

Browser tooling и HTTP tooling должны поддерживать deferred credential references вместо передачи секретов literal values.

Базовые требования:
- пользователь работает с credential aliases;
- агент видит только alias и metadata / capability hints;
- tool args могут содержать typed `credential_ref` объекты;
- фактическое разрешение секрета выполняется только в системном слое перед исполнением tool runtime;
- tool outputs и логи не должны возвращать реальные секретные значения.

## 15. Docker-oriented execution model

### 15.1 Базовый принцип

Все части Butler должны быть ориентированы на контейнерное развертывание.

### 15.2 Принятое решение

Принята стратегия **container per tool class (контейнер на класс инструментов)** — см. Tool Runtime ADR (`docs/architecture/tool-runtime-adr.md`, секция 9).

Рассматривались два варианта:

**Вариант A:** один общий tool runtime container (контейнер выполнения инструментов) для нескольких tools  
**Вариант B:** отдельный container per tool / tool class (контейнер на каждый инструмент или класс инструмента)

### 15.3 Анализ вариантов

#### Вариант A — общий контейнер

Плюсы:
- проще запуск и сопровождение;
- меньше накладных расходов;
- проще MVP;
- меньше orchestration complexity (сложности оркестрации).

Минусы:
- слабее изоляция;
- выше риск смешивания зависимостей;
- хуже security boundary (граница безопасности);
- труднее ограничивать ресурсы по отдельным tool classes.

#### Вариант B — контейнер на tool class

Плюсы:
- лучше изоляция;
- лучше контроль зависимостей;
- проще лимитировать ресурсы;
- проще ввести разные security policies.

Минусы:
- выше сложность orchestration;
- больше контейнеров;
- больше межсервисного взаимодействия;
- выше operational complexity.

### 15.4 Рекомендация

Для V1 рекомендуется **не делать контейнер на каждый отдельный tool**, но и не сваливать всё в один полностью общий runtime.

Рекомендуемая компромиссная модель:
- один контейнер для browser automation;
- один контейнер для HTTP / web;
- один контейнер для self-inspection / doctor;
- при необходимости отдельный orchestrator / broker container.

То есть базовая стратегия Butler V1: **container per tool class (контейнер на класс инструментов)**.

Это даёт значительно лучшую изоляцию, чем единый runtime, но остаётся проще, чем модель “отдельный контейнер на каждый tool instance”.

## 16. Подсистема управления учётными данными и отложенного разрешения секретов

Подсистема credential management является обязательной частью Butler V1 для сценариев, где agent-driven automation требует авторизации, но модель не должна видеть секретные значения.

### 16.1 Базовый принцип

Пользователь оперирует credential alias, а не секретом. Агент получает только alias, тип секрета, допустимые target hints и допустимые способы использования. Реальные значения разрешаются только в runtime через Credential Broker.

### 16.2 Ключевые компоненты

- **Credential Alias** — пользовательский идентификатор набора учётных данных;
- **Credential Record** — metadata-запись с allowed domains, allowed tools и approval policy;
- **Deferred Credential Reference** — типизированная ссылка вида `credential_ref`, передаваемая в tool args;
- **Credential Broker** — системный слой разрешения ссылок и аудита;
- **Secret Store** — отдельное хранилище фактических секретов.

### 16.3 Требования к архитектуре

- агент не должен видеть пароль, токен, cookie value или storage state;
- обычный model context не должен содержать secret material;
- browser tools и HTTP tools должны использовать единый механизм авторизации через alias и `credential_ref`;
- доступ к alias должен ограничиваться allowed domains, allowed tools, approval policy и autonomy mode;
- каждое использование alias должно логироваться.

### 16.4 Режимы и подтверждение

Credential subsystem должна учитывать как approval policy конкретного alias, так и глобальный autonomy mode Butler. В read-only режимах использование alias должно быть запрещено или требовать явного подтверждения в зависимости от policy.

---

## 17. Tool Broker

Tool Broker — обязательный архитектурный компонент.

### 17.1 Ответственность Tool Broker

- принимать tool call requests;
- валидировать параметры;
- проверять режим автономности и policy rules;
- маршрутизировать вызов к нужному tool runtime container;
- взаимодействовать с Credential Broker при наличии `credential_ref` в аргументах;
- логировать выполнение;
- возвращать нормализованный tool result.

### 17.2 Почему это важно

Модель не должна иметь прямой доступ к исполнителям инструментов. Между моделью и исполнением нужен контрольный слой.

---

## 18. Режимы автономности

В Butler должна быть заложена поддержка режимов автономности.

### 18.1 Предварительная модель режимов

#### Mode 0 — Read / Assist
- агент отвечает и анализирует;
- не выполняет изменяющие действия.
- использование credential alias запрещено.

#### Mode 1 — Propose
- агент может предлагать действия;
- выполнение происходит только после подтверждения пользователя.
- использование credential alias требует подтверждения.

#### Mode 2 — Guarded Execute
- агент может выполнять часть разрешённых действий самостоятельно;
- опасные действия требуют подтверждения.
- credential use разрешается только при совместимости с alias policy и domain restrictions.

#### Mode 3 — Extended Autonomy
- агент может выполнять широкий спектр действий в пределах явно заданной политики.
- credential use определяется policy, domain restrictions и tool scope.

### 18.2 Требование к архитектуре

Режим автономности должен быть отдельным policy input (входным параметром политики), а не hardcoded behavior (жёстко зашитым поведением).

---

## 19. Self-inspection / Doctor subsystem

Butler должен уметь помогать пользователю настраивать самого себя.

### 19.1 Назначение

Подсистема doctor должна проверять:
- здоровье контейнеров через health endpoints и другие безопасные диагностические интерфейсы;
- доступность БД и Redis;
- доступность model providers;
- корректность конфигурации каналов;
- состояние памяти и индексов;
- состояние tool runtimes;
- версию и конфигурацию самой системы.

### 19.2 Выходные данные

Doctor должен уметь формировать:
- краткий статус;
- список проблем;
- объяснение причин;
- рекомендации по исправлению;
- при возможности — безопасные полуавтоматические действия.

Полуавтоматические действия в этой модели означают рекомендацию или вызов отдельного ограниченного control-plane сервиса.
Doctor не должен сам исполнять Docker lifecycle operations.

### 19.3 Особая ценность

Наличие встроенного self-inspection слоя делает Butler не просто агентом, а самообслуживаемой системой, что особенно важно для self-hosted продукта.

---

## 20. Безопасность

### 20.1 Принципы

1. Минимально необходимые права.
2. Изоляция execution surfaces.
3. Явное разделение read-only и mutating operations.
4. Подтверждение пользователя для опасных действий.
5. Полный audit trail.
6. Секреты не попадают в модельный контекст.

### 20.2 Минимальные меры V1

- контейнерная изоляция по классам инструментов;
- отдельный Tool Broker;
- отдельный Credential Broker и Secret Store;
- логирование всех tool calls;
- аудит всех обращений к credential aliases;
- режимы автономности;
- ограничения по сетевому доступу там, где это возможно;
- маскирование логов и tool outputs при работе с секретами;
- ограничения по ресурсам контейнеров.

---

## 21. Наблюдаемость и эксплуатация

Система должна предоставлять:
- логи компонентов;
- статус контейнеров;
- таймлайны выполнения run;
- журнал tool calls;
- журнал использования credential aliases;
- статистику использования моделей;
- диагностику памяти;
- отчёты doctor.

Web UI должен стать основной точкой эксплуатационной наблюдаемости.

## 25. Итоговый архитектурный тезис

Butler (дворецкий) проектируется как self-hosted, docker-oriented, personal long-lived agent platform (персональная долгоживущая агентная платформа), где:
- WebSocket-first transport ускоряет и упрощает взаимодействие с моделями;
- память остаётся собственной подсистемой Butler;
- инструменты исполняются через контролируемый broker и контейнерные runtimes;
- секреты разрешаются отложенно через alias, Credential Broker и Secret Store без раскрытия агенту;
- система умеет не только помогать пользователю, но и объяснять собственное состояние и помогать себя настраивать.

Это делает Butler не клоном обычного chat agent (чат-агента), а архитектурно самостоятельной агентной системой с упором на долговременность, управляемость и self-hosted эксплуатацию.
