# Butler — Technology Stack and Engineering Standards

## 1. Статус документа

**Тип документа:** Engineering Standards + Technology Stack  
**Версия:** 1.0  
**Статус:** Draft  
**Целевая аудитория:** AI-агенты, работающие с репозиторием Butler

---

## 2. Назначение

Этот документ фиксирует:
- целевой стек технологий Butler;
- правила выбора технологий;
- стиль разработки;
- инженерные best practices;
- ограничения по зависимостям;
- минимальный обязательный quality baseline.

Документ обязателен для AI-агентов, вносящих изменения в репозиторий Butler.

---

## 3. Ключевые инженерные принципы

### 3.1 Minimal dependencies first

Butler должен использовать минимально возможное количество внешних зависимостей.

Каждая новая зависимость должна быть обоснована по одному из критериев:
1. существенно сокращает сложность реализации;
2. закрывает критичную инфраструктурную задачу;
3. даёт проверенную промышленную реализацию вместо самодельной;
4. снижает риск ошибок и упрощает сопровождение.

Если задача может быть решена стандартной библиотекой или уже принятым dependency set (набором зависимостей), новая библиотека добавляться не должна.

### 3.2 Go-first policy

Во всех возможных случаях для основной системы и tool runtime используется Go.

Python допускается только в исключительных случаях, если одновременно выполняются условия:
- реализация на Go нерационально сложна;
- существует зрелая и практически необходимая Python-экосистема;
- изоляция такого компонента сохраняется;
- использование Python не создаёт архитектурной зависимости всей системы от Python runtime.

### 3.3 Self-hosted and Docker-oriented by default

Любое решение должно быть совместимо с self-hosted моделью Butler и с развёртыванием через Docker Compose.

### 3.4 Stability over novelty

Для Butler предпочтительны простые, проверенные и сопровождаемые решения. Экзотические технологии и сложные платформенные решения без крайней необходимости не допускаются.

### 3.5 AI-agent friendliness

Кодовая база должна быть максимально понятной и предсказуемой для AI-агентов:
- явные интерфейсы;
- минимальная магия;
- низкая неявность;
- ограниченное число паттернов;
- короткие и понятные зависимости между модулями.

## 4. Целевая архитектурная модель разработки

### 4.1 Архитектурный стиль

Butler проектируется как microservices architecture (микросервисная архитектура) с ограниченным числом сервисов, без искусственного дробления.

### 4.2 Практическая интерпретация

В Butler микросервисность означает:
- отдельные сервисы для действительно разных обязанностей;
- чёткие API-контракты;
- независимое контейнерное развертывание;
- отдельные runtime для tool classes.

Микросервисность **не означает**:
- дробление на десятки мелких сервисов без реальной необходимости;
- преждевременное усложнение оркестрации;
- дублирование инфраструктурного кода.

### 4.3 Рекомендуемые сервисные границы V1

Минимально разумное разбиение:
- orchestrator / core API service;
- tool broker;
- browser runtime;
- http runtime;
- doctor runtime;
- web UI;
- PostgreSQL;
- Redis.

## 5. Технологический стек

## 5.1 Backend

### Основной язык
- **Go**

### Причины выбора
- высокая предсказуемость runtime;
- хорошая пригодность для сетевых сервисов и контейнерной эксплуатации;
- малая операционная сложность;
- сильная стандартная библиотека;
- удобство сборки и поставки;
- хороший баланс производительности и простоты.

### Python policy
Python разрешён только для изолированных компонентов, где это действительно необходимо.

Примеры допустимого применения:
- узкоспециализированная интеграция, если в Go нет зрелой альтернативы;
- отдельные data-processing / parsing сценарии, если выгода очевидна;
- экспериментальные инструменты вне core path.

Python не должен использоваться как основной язык orchestrator, broker или базовой платформенной логики.

---

## 5.2 Frontend

### Выбранный стек
- **Vue**
- **Nuxt.js**

### Назначение
Web UI Butler используется для:
- наблюдаемости;
- конфигурации;
- просмотра памяти;
- управления режимами;
- doctor reports.

### Ограничения
- избегать избыточных UI-framework зависимостей;
- использовать встроенные возможности Nuxt.js там, где этого достаточно;
- минимизировать клиентскую сложность.

---

## 5.3 API и межсервисное взаимодействие

### Внешний API
- **REST** для простого внешнего взаимодействия и Web UI backend flows

### Внутренние сервисные контракты
- **gRPC** для межсервисного взаимодействия, где важны строгие контракты и typed schemas

### Правило выбора
- внешний пользовательский и административный API: REST;
- внутренние service-to-service вызовы: gRPC;
- не смешивать оба стиля без необходимости на одном и том же интерфейсе.

---

## 5.4 Хранилища

### Основное хранилище
- **PostgreSQL**

### Векторный слой
- **pgvector**

### Краткоживущий state layer
- **Redis**

### Правило
PostgreSQL является source of truth (источником истины) для системных и memory данных.  
pgvector используется как retrieval layer (слой извлечения), а не как самостоятельная память.  
Redis используется только для short-lived state (краткоживущего состояния), locks и caches.

---

## 5.5 Очереди и асинхронное выполнение

### Выбранное решение для Butler V1
- **Redis-based queue**

### Причины выбора
- минимизация инфраструктурной сложности;
- уже используемый Redis в системе;
- достаточность для V1;
- простота self-hosted эксплуатации через Docker Compose.

### Ограничения
Kafka, RabbitMQ или NATS не вводятся в V1 без доказанной необходимости.

### Когда пересматривать решение
Отдельный message broker должен рассматриваться только если появятся:
- тяжёлые фоновые пайплайны;
- высокая конкуренция задач;
- необходимость сложной event-driven оркестрации;
- требования по более сложной гарантии доставки.

---

## 5.6 Контейнеризация и окружение

### Базовая модель
- **Docker Compose only** для основной эксплуатационной модели

### Правила
- каждый сервис должен иметь собственный Dockerfile;
- compose-конфигурация должна быть первичным способом локального и серверного запуска;
- все сервисы должны стартовать предсказуемо в self-hosted окружении;
- не закладываться на Kubernetes как обязательную часть архитектуры.

---

## 6. Рекомендации по библиотекам

Цель — минимальное число зависимостей.

## 6.1 Go backend

### HTTP / REST
Предпочтение:
- стандартная библиотека `net/http`

Допустимо:
- лёгкий router только если действительно нужен и упрощает код

### gRPC
- официальный `grpc-go`

### Конфигурация
Предпочтение:
- env + собственный тонкий config layer

Не рекомендуется:
- тяжёлые config frameworks без крайней необходимости

### Валидация
Предпочтение:
- простая ручная валидация для core config и request DTO
- protobuf / schema-first валидация там, где это уже обеспечивается контрактом

### Логирование
Предпочтение:
- стандартный structured logging подход на основе минимального набора
- использовать один единый logging package на весь backend

### Работа с PostgreSQL
Предпочтение:
- `pgx`

### Работа с Redis
Предпочтение:
- минимальный зрелый Redis client

### Browser automation
Выбор должен быть отложен до отдельного ADR, но базовое правило:
- использовать один основной browser runtime adapter;
- не поддерживать несколько engine implementations одновременно без необходимости.

### HTTP client
Предпочтение:
- стандартный `net/http` client с собственной thin abstraction.

---

## 6.2 Frontend

### Обязательный минимум
- Nuxt.js
- Vue

### Ограничения
- не добавлять state-management framework, если хватает встроенных средств;
- избегать тяжёлых UI kit, если можно обойтись простыми компонентами;
- не строить frontend как отдельную сложную платформу.

## 7. Стиль разработки

## 7.1 General style

Для Butler выбирается стиль:
- строгая структурность;
- простые модули;
- явные интерфейсы;
- ограниченный размер файлов и пакетов;
- минимальный уровень абстракций.

## 7.2 Код должен быть понятен AI-агенту

Это означает:
- избегать скрытых side effects;
- избегать reflection-heavy кода там, где это не нужно;
- избегать глубокой метапрограммности;
- не использовать сложные framework conventions как основу архитектуры.

## 7.3 Предпочтение explicit over implicit

Лучше:
- явные структуры данных;
- явные контракты;
- явные зависимости через интерфейсы.

Хуже:
- неявные глобальные состояния;
- скрытая инициализация;
- magic wiring.

---

## 8. Структура репозитория

Рекомендуемая модель:
- monorepo (монорепозиторий) с несколькими сервисами

### Причины
- проще для AI-агентов;
- проще для единых стандартов;
- проще для общей документации;
- проще для Docker Compose deployment.

### Рекомендуемый каркас
- `apps/` — сервисы
- `packages/` или `internal/` — общие библиотеки
- `docs/` — документация
- `docker/` или `deploy/` — контейнерные артефакты
- `.opencode/` — команды и инструкции для AI-агентов

---

## 9. Тестовая стратегия

### 9.1 Базовый принцип
Тесты обязательны, но без избыточного тестового шума.

### 9.2 Обязательный минимум
1. **Unit tests** для:
   - бизнес-логики;
   - policy logic;
   - config parsing;
   - memory extraction;
   - credential resolution;
   - broker validation logic.

2. **Integration tests** для:
   - PostgreSQL integration;
   - Redis integration;
   - gRPC / REST contracts;
   - tool broker ↔ runtime interactions.

3. **Selective e2e tests** только для критических сценариев:
   - базовый пользовательский flow;
   - browser tool flow;
   - doctor diagnostics flow;
   - credential-aware execution flow.

### 9.3 Что не требуется в baseline
- широкие snapshot-heavy тесты;
- массовые e2e тесты на всё подряд;
- искусственное покрытие ради процента coverage.

### 9.4 Правило
Лучше меньше тестов, но с реальной ценностью для архитектуры и безопасности.

## 10. Документация

### Обязательное требование
Каждый сервис должен иметь свой `README.md`.

### В README сервиса должно быть минимум:
- назначение;
- основные зависимости;
- конфигурация;
- точки входа;
- как запускать локально;
- как тестировать;
- какие API он предоставляет;
- какие документы архитектуры к нему относятся.

### Дополнительно
Если сервис архитектурно чувствителен, должен существовать отдельный subspec в `docs/architecture/`.

---

## 11. Observability

### Обязательный baseline
1. **Structured logs** для всех сервисов
2. **Минимальные метрики**

### Логи
Все сервисы обязаны писать структурированные логи с единым форматом.

Минимальные поля:
- `service`
- `component`
- `run_id` при наличии
- `tool_call_id` при наличии
- `level`
- `message`
- `timestamp`

### Метрики
Минимально необходимо собирать:
- количество запросов;
- количество ошибок;
- длительность операций;
- количество tool calls;
- количество credential-aware tool invocations;
- количество doctor checks.

### Что не требуется в baseline
- полноценная распределённая трассировка;
- сложные observability платформы.

---

## 12. Security baseline

Для Butler безопасность должна быть встроенной частью инженерных стандартов.

### Обязательно
1. **Secret management**  
   секреты не должны попадать в модельный контекст, логи и обычные tool outputs.

2. **Least privilege**  
   каждый сервис и runtime должен иметь только минимально необходимые права.

3. **Container isolation**  
   tool runtimes должны быть изолированы по классам инструментов.

4. **Audit logging**  
   все чувствительные действия должны логироваться на уровне аудита.

5. **Approval flows**  
   изменяющие и рискованные действия должны учитывать режимы автономности и policy layer.

6. **Dependency restraint**  
   каждая новая зависимость рассматривается как security surface.

### Желательно
- регулярная проверка зависимостей;
- явная маркировка секретных config keys;
- masking чувствительных значений во всех diagnostics и doctor reports.

## 13. Работа AI-агентов с репозиторием

Так как документ ориентирован на AI-агентов, правила работы обязательны.

### 13.1 Перед изменениями агент обязан
1. прочитать релевантные документы из `docs/architecture/`;
2. проверить `AGENTS.md`;
3. не менять архитектурные границы без явного обновления документации;
4. сохранять совместимость с Docker-oriented моделью;
5. не вводить лишние зависимости.

### 13.2 AI-агенту запрещено
- добавлять тяжёлую библиотеку без обоснования;
- смешивать transport, memory и tool execution в один слой;
- вводить скрытые глобальные зависимости;
- обходить policy / broker слой;
- передавать секреты в модельный контекст.

### 13.3 AI-агент должен
- предпочитать простые решения;
- сохранять читаемость для следующего AI-агента;
- обновлять README сервиса, если меняется поведение сервиса;
- держать контракты стабильными.

---

## 14. Стандарты проектирования кода

### 14.1 Предпочтения
- короткие пакеты с одной ответственностью;
- явные интерфейсы на границах сервиса;
- DTO / transport models отдельно от domain models;
- минимальная вложенность;
- минимальные side effects.

### 14.2 Не рекомендуется
- god objects;
- чрезмерное количество слоёв ради “чистой архитектуры” без практической пользы;
- framework-driven проектирование;
- чрезмерное использование generic abstractions.

### 14.3 Приоритет
Код Butler должен быть проще поддерживать, чем “красиво абстрагировать”.

---

## 15. Подход к зависимостям

### 15.1 Перед добавлением новой зависимости нужно проверить
1. можно ли решить стандартной библиотекой;
2. можно ли решить уже существующими зависимостями;
3. насколько библиотека зрелая;
4. насколько велик её dependency tree;
5. как она влияет на контейнерный образ и сопровождение.

### 15.2 Разрешение на зависимость
Новая зависимость допустима только если она:
- ощутимо упрощает код;
- уменьшает риск ошибок;
- не тянет тяжёлый стек без необходимости;
- соответствует self-hosted и Docker-oriented модели.

---

## 16. Качество как приоритет

Для Butler в начальной стадии важнее:
- надёжность;
- предсказуемость;
- архитектурная чистота в практическом смысле;
- сопровождаемость;
- безопасность.

Скорость разработки важна, но не за счёт долгосрочной деградации архитектуры.

---

## 17. Итоговый технологический выбор

### Обязательный стек Butler V1
- Backend: **Go**
- Frontend: **Vue + Nuxt.js**
- External API: **REST**
- Internal API: **gRPC**
- Database: **PostgreSQL**
- Vector layer: **pgvector**
- Short-lived state: **Redis**
- Async jobs: **Redis-based queue**
- Deployment: **Docker Compose**

### Языковая политика
- Go по умолчанию
- Python только по исключению

### Инженерная политика
- минимальное количество зависимостей;
- обязательные README на каждый сервис;
- структурированные логи;
- минимальные полезные метрики;
- обязательные unit + integration tests;
- ограниченные, но обязательные e2e для критических сценариев;
- сильный security baseline.

---

## 18. Финальный тезис

Butler должен разрабатываться как self-hosted, Docker-oriented, Go-first система с минимальным dependency footprint (следом зависимостей), строгими архитектурными границами, обязательной документацией на каждый сервис и предсказуемым стилем разработки, удобным не только для людей, но и для AI-агентов.

Главное правило разработки Butler:

**простота, явность, минимализм зависимостей и архитектурная дисциплина важнее технологической моды и избыточной сложности.**
