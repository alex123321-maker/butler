# Butler Web UI — AI-ready Frontend Specification

## 1. Назначение документа

Этот документ фиксирует единую спецификацию для разработки нового web UI (веб-интерфейса) Butler в monorepo (монорепозитории).

Документ должен использоваться как основной источник правды для:
- проектирования фронтенд-архитектуры
- настройки дизайн-системы
- организации agent-driven development (разработки с ИИ-агентами)
- описания структуры страниц и ожидаемого контента
- ограничения UI-решений, чтобы web UI не превратился во второй канал общения с агентом

Документ предназначен для AI-кодеров, работающих через Opencode (Опенкод) с провайдерами Codex (Кодекс) и Copilot (Копайлот), и для человека, который проверяет архитектурную целостность решений.

---

## 2. Продуктовая модель

### 2.1. Основная формула продукта

Butler — это система, где:
- пользователь ставит задачи через Telegram (Телеграм)
- ИИ-агенты исполняют задачи
- web UI используется для наблюдения, контроля, подтверждений, результатов, ошибок и памяти

### 2.2. Telegram-first модель

Нужно считать жёстко закреплённым следующее:
- Telegram — единственная точка входа пользовательских задач
- web UI не используется для создания задач
- web UI не является assistant interface (интерфейсом ассистента)
- web UI является operational control console (операционной консолью контроля и прозрачности)

### 2.3. Главные сущности UI

UI должен строиться вокруг следующих доменных сущностей:
- Task / Run (задача / выполнение)
- Approval (подтверждение)
- Artifact (артефакт результата)
- Memory (память)
- Activity (аудит и действия)
- System State (состояние платформы)

### 2.4. Жёсткие продуктовые запреты

В web UI запрещено добавлять:
- поле отправки сообщения агенту
- кнопку создания задачи
- prompt box (поле промпта)
- активный рабочий чат
- любые UX-паттерны, предполагающие запуск задач из браузера
- scheduler (планировщик)
- periodic tasks (периодические задачи)
- cron builder (конструктор cron)

## 3. Цели фронтенд-архитектуры

Новая фронтенд-система должна:
- быть понятной AI-агентам и человеку
- быть устойчивой к постепенному росту количества экранов и сущностей
- позволять быстро добавлять новые продуктовые блоки без размывания структуры
- поддерживать строгую визуальную и кодовую консистентность
- быть пригодной для test-first (подхода «сначала тесты»)
- позволять работать без моков, сразу с реальным backend (бэкендом), при условии поэтапной синхронизации API

Ключевая идея: фронтенд проектируется не как набор страниц, а как доменная система состояния и наблюдения.

## 4. Технический стек

### 4.1. Базовый стек

Фиксируется следующий стек:
- Nuxt (Нукст)
- Vue 3 (Вью 3)
- TypeScript (ТайпСкрипт)
- Pinia (Пиния) для state management (управления состоянием)
- Tailwind CSS (Тейлвинд CSS)
- PrimeVue (ПраймВью) как базовая UI-библиотека
- Lucide (Люсид) для иконок
- Playwright (Плейрайт) для e2e (сквозных) и visual regression (визуальных регрессионных) тестов
- ESLint (ЕСЛинт) + Prettier (Приттир) + Stylelint (Стайллинт)

### 4.2. Режим приложения

Приложение должно быть реализовано как:
- internal SPA (внутреннее одностраничное приложение)
- на базе Nuxt (Нукст)
- с отключённым SSR (серверным рендерингом) для основного продуктового режима

Причины:
- продукт не зависит от SEO
- это внутренняя control console (консоль управления)
- доминирует интерактивная работа со статусами, списками и системным состоянием

### 4.3. Адаптивность

Базовый режим:
- desktop-first (сначала десктоп)
- tablet-friendly (пригодно для планшета)
- mobile (мобильный режим) допускается только в упрощённом виде

Это значит:
- основной UX проектируется под широкие экраны
- сложные таблицы и detail layout (детальные раскладки) не обязаны полноценно ужиматься до полноценного mobile-first формата
- мобильная версия не должна становиться отдельным продуктом

### 4.4. Обновление данных

На первом этапе используем:
- SSE (Server-Sent Events — серверные push-события) для обновления критичных состояний
- ручное обновление как fallback (резервный режим)

SSE должен применяться как минимум для:
- active tasks (активных задач)
- approvals (подтверждений)
- activity (активности)
- system state (состояния системы)

Если backend (бэкенд) ещё не готов к SSE, допускается временный polling (опрос), но в спецификации интерфейса продуктовая модель остаётся событийной.

## 5. Архитектурный подход

### 5.1. Основной подход

Используется Feature-Sliced Design (функционально-срезанная архитектура).

Цель:
- разделить код по бизнес-сущностям и сценариям
- не допустить хаоса вида `components/ + views/ + utils/`
- сделать структуру проекта предсказуемой для AI-кодера

### 5.2. Базовая структура проекта

```text
  web/
    app/
    pages/
    widgets/
    features/
    entities/
    shared/
```

### 5.3. Правила слоёв

#### app
Содержит только:
- глобальные providers (провайдеры)
- плагины
- корневые конфигурации
- регистрацию тем
- регистрацию PrimeVue (ПраймВью)

#### pages
Содержит:
- композицию страницы
- раскладку верхнего уровня
- подключение widgets (виджетов)
- без тяжёлой бизнес-логики

#### widgets
Содержит:
- большие визуальные блоки страниц
- сборку нескольких features (фич) и entities (сущностей)

#### features
Содержит:
- пользовательские действия и сценарии
- логику одного конкретного сценария

#### entities
Содержит:
- доменные типы
- модель представления сущности
- API-адаптеры сущности
- селекторы, форматтеры и presentation helpers (помощники отображения)

#### shared
Содержит:
- переиспользуемый UI
- utility-функции
- дизайн-токены
- общие хуки и composables (композиции)
- API-клиент

## 6. Правила проектирования кода

### 6.1. Общие правила

Каждый блок должен быть:
- максимально детерминированным
- машиночитаемым
- предсказуемым
- изолированным по ответственности

### 6.2. Правила компонентов

Компоненты должны:
- иметь одну чёткую ответственность
- не смешивать API-вызовы, сложную бизнес-логику и рендер в одном файле
- быть по возможности dumb components (тупыми компонентами), если это чистый UI

Компоненты не должны:
- напрямую знать структуру backend API (бэкенд API), если это можно вынести в слой `shared/api` или `entities/*/api`
- содержать произвольный CSS вне общей системы
- импортировать PrimeVue (ПраймВью) напрямую за пределами `shared/ui`

### 6.3. Composition API

Используется только современный стиль:
- `<script setup>`
- Composition API (композиционный API)
- composables (композиции) для повторного поведения

### 6.4. TypeScript

TypeScript обязателен для:
- API response types (типов ответов API)
- UI view models (моделей представления)
- props (свойств компонентов)
- forms (форм)
- status maps (карт статусов)
- filters (фильтров)

Использование `any` запрещено, кроме явно документированных переходных мест.

### 6.5. State management

Pinia используется как центральный слой состояния для:
- глобальных фильтров
- данных текущего пользователя
- системного состояния
- кэша активных разделов
- SSE subscriptions (SSE-подписок), если они централизованы

Локальное состояние допускается в компонентах только для:
- UI toggles (локальных переключателей)
- controlled inputs (контролируемых полей)
- временных раскрытий и панелей

### 6.6. Работа с API

Работа с REST (РЕСТ API) должна быть организована так:
- `shared/api/client` — общий HTTP client (HTTP-клиент)
- `entities/*/api` — доменные адаптеры
- `features/*/model` — orchestration logic (логика сценария)
- `widgets` и `pages` — не знают деталей запросов

Моки не являются основным режимом. Фронт должен ориентироваться на реальный backend (бэкенд), а расхождения между новым UI и текущим API надо фиксировать отдельно в backend change spec (спецификации изменений бэкенда).

## 7. Design system (дизайн-система)

### 7.1. Общая идея

UI не должен собираться стихийно. Он должен строиться из:
- дизайн-токенов
- ограниченного набора базовых компонентов
- доменных шаблонов отображения

Задача дизайн-системы:
- исключить произвольные визуальные решения
- упростить работу AI-кодера
- сохранить единый характер продукта

### 7.2. Визуальный характер

Интерфейс должен быть:
- минималистичным
- монохромным по базе
- тёмным по умолчанию
- спокойным
- профессиональным
- ближе к control console (операционной консоли), чем к marketing dashboard (маркетинговой панели)

Не использовать:
- неоновые акценты
- футуристический «AI-стиль»
- декоративную перегруженность
- chat-first паттерны
- стеклянные и эффектные визуальные решения ради декора

### 7.3. Тема

На первом этапе фиксируется только одна тема:
- dark theme (тёмная тема)

Light theme (светлая тема) не входит в MVP (минимально жизнеспособный продукт).

### 7.4. Цветовая система

База интерфейса — монохромная:
- несколько оттенков почти чёрного
- несколько оттенков серого
- белый и off-white (мягкий белый)

Разрешены:
- один основной акцентный цвет для выделения активности, ссылок, фокуса и selected state (состояния выбора)
- отдельный красный диапазон для ошибок, риска и destructive actions (разрушающих действий)
- мягкие статусные оттенки для success (успеха) и warning (предупреждения), но очень умеренно

### 7.5. Рекомендованная токен-модель


### 7.6. Spacing tokens (токены отступов)

Использовать единую шкалу:

```text

space.4 = 16px
space.5 = 20px
```

### 7.7. Radius tokens (токены скруглений)

```text
radius.sm = 6px
radius.md = 10px
radius.lg = 14px
radius.xl = 18px
```

### 7.8. Shadow tokens (токены теней)

Тени должны быть минимальными, так как продукт тёмный и серьёзный.

Допускаются только:
- слабая тень для elevated surfaces (поднятых поверхностей)
- подчёркивание границ через border (границу) важнее, чем через тень

### 7.9. Типографика

Рекомендованный стек:
- основной интерфейс: `IBM Plex Sans (Ай-Би-Эм Плекс Санс)`
- технические значения, время, ID, debug, фрагменты Telegram и метки: `IBM Plex Mono (Ай-Би-Эм Плекс Моно)`

Почему:
- есть техничность
- есть ощущение консоли
- при этом интерфейс не выглядит устаревшим и слишком жёстким

### 7.10. Плотность

Плотность интерфейса:
- средняя
- с уклоном в компактную

Следствия:
- таблицы должны быть плотными, но читаемыми
- карточки не должны быть избыточно крупными
- вертикальные отступы должны быть экономными
- текстовая иерархия важнее пустого воздуха

## 8. Design tokens (дизайн-токены)

### 8.1. Принцип

Все визуальные решения должны опираться на tokens (токены). Хардкод значений запрещён.

Запрещено:
- hex colors (hex-цвета) напрямую
- inline spacing (встроенные отступы)
- произвольные border radius (радиусы скругления)
- случайные opacity values (значения прозрачности)

### 8.2. Источник токенов

Токены должны храниться централизованно:

```text
shared/design/tokens/
  colors.ts
  spacing.ts
  typography.ts
  radius.ts
  shadows.ts
  semantic.ts
```

Дополнительно допускается генерация CSS variables (CSS-переменных):

```text
app/styles/tokens.css
```

### 8.3. Tailwind integration (интеграция с Tailwind)

Tailwind должен быть расширен из токенов, а не использоваться как свободный набор утилит.

Разрешено:
- использовать utility-классы только из согласованной токенизированной системы
- описывать theme extension (расширение темы) в конфиге Tailwind

Запрещено:
- произвольные цвета через квадратные скобки
- произвольные padding/margin значения
- случайные z-index значения без токенов

## 9. PrimeVue (ПраймВью) и базовые UI-компоненты

### 9.1. Стратегия использования

PrimeVue используется как основа, но не как прямой публичный API проекта.

Правило:
- PrimeVue импортируется только в `shared/ui`
- в остальных слоях используются только внутренние обёртки проекта

Это нужно, чтобы:
- не размывать визуальную систему
- не тащить библиотечные решения напрямую во все слои
- иметь возможность заменить библиотеку или глубоко переоформить её позже

### 9.2. Базовые primitives (примитивы)

На старте нужно создать минимум следующие базовые обёртки:
- `AppButton`
- `AppCard`
- `AppTable`
- `AppBadge`
- `AppPanel`
- `AppDialog`
- `AppInput`
- `AppTextarea`
- `AppSelect`
- `AppTabs`
- `AppEmptyState`
- `AppSkeleton`
- `AppAlert`
- `AppTooltip`
- `AppIconButton`

### 9.3. Правила создания новых компонентов

Разрешено:
- создавать новые составные компоненты в `widgets` и `features`
- создавать новый базовый primitive (примитив) только если его ещё нет в системе

Запрещено:
- создавать дубли базовых кнопок, бейджей, карточек, таблиц и диалогов, если в системе уже есть рабочая версия

### 9.4. API базовых компонентов

Базовые компоненты должны иметь ограниченный и предсказуемый API.

Например, для кнопки:
- `variant`
- `size`
- `tone`
- `loading`
- `disabled`
- `icon`

Запрещено:
- пробрасывать произвольный inline style (встроенный стиль)
- пробрасывать сырой набор визуальных параметров, который ломает единообразие

## 10. Иконки

Единая система иконок:
- Lucide (Люсид)

Правила:
- не смешивать Lucide (Люсид) с PrimeIcons (ПраймАйконс) и другими наборами
- иконки использовать функционально, а не декоративно
- статус иконкой усиливает смысл, но не заменяет текст

## 11. Машиночитаемость UI для AI-агентов

### 11.1. Почему это важно

AI-кодер работает лучше, когда интерфейс:
- собран из ограниченного набора примитивов
- описан доменными сущностями
- имеет явные состояния
- не требует «угадывать» стили

### 11.2. Обязательные требования

Каждый сложный экран должен иметь:
- явную иерархию блоков
- устойчивые `data-testid` атрибуты для критичных элементов
- доменные названия компонентов
- предсказуемые loading / empty / error states (состояния загрузки / пустоты / ошибки)

### 11.3. Запрещённые практики

Запрещено:
- именовать компоненты слишком абстрактно (`Box`, `Panel2`, `CustomItem`)
- скрывать важные состояния только через цвет
- строить сложный layout (макет) на случайных div без смысла
- создавать одноразовые стили, которые не ложатся в дизайн-систему

## 12. Multi-agent workflow (многоагентный процесс) для Opencode

### 12.1. Базовые роли

Процесс разработки должен быть совместим с Opencode (Опенкод) и строиться вокруг трёх ролей:

#### 1. Architect agent (агент-архитектор)
Отвечает за:
- разбор задачи
- определение слоёв FSD (функционально-срезанной архитектуры)
- список файлов
- влияние на API
- ограничения дизайн-системы
- критерии готовности

#### 2. Builder agent (агент-реализатор)
Отвечает за:
- реализацию компонентов
- подключение API
- сборку состояния
- интеграцию со страницей
- соблюдение токенов и базовых primitives (примитивов)

#### 3. QA agent (агент-тестировщик)
Отвечает за:
- Playwright tests (тесты Playwright)
- smoke tests (базовые тесты загрузки)
- critical path tests (тесты критического сценария)
- visual regression (визуальную регрессию)
- проверку доступности ключевых сценариев

### 12.2. Правила взаимодействия агентов

Каждая новая задача должна проходить стадии:
1. Architect agent описывает план изменений
2. QA agent формулирует ожидаемые проверки и test cases (тест-кейсы)
3. Builder agent реализует код
4. QA agent запускает и корректирует тесты

### 12.3. Обязательное требование

Ни один агент не должен:
- придумывать новую визуальную систему
- нарушать Telegram-first ограничения
- создавать свободный чатовый UX
- обходить `shared/ui`

## 13. Test-first стратегия

### 13.1. Общий принцип

Новые страницы и критичные функции должны разрабатываться по модели:
- сначала описывается ожидаемое поведение
- затем пишутся Playwright tests (тесты Playwright)
- затем реализуется UI

### 13.2. Минимальный набор тестов для каждой новой страницы

Обязательно:
- smoke test — страница открывается и рендерит ключевой блок
- navigation test — срабатывает базовая навигация и переходы
- critical state test — проверяются ключевые состояния страницы
- screenshot test — фиксируется эталонный визуальный снимок

### 13.3. Что считать критичными состояниями

Например:
- loading
- empty
- error
- waiting for approval
- waiting for Telegram reply
- completed result
- failed state

### 14.1. Назначение

Playwright используется как:

## 16. Правила контента и UX

### 16.1. Что всегда показывать

### 16.3. Telegram visibility (видимость Telegram)

На Task / Run экранах обязательно показывать:
- что задача пришла из Telegram
- исходное сообщение
- был ли отправлен ответ обратно
- ожидается ли сейчас ответ пользователя именно в Telegram

## 29. Data contracts (контракты данных) и backend coordination (координация с бэкендом)

### 29.1. Общий подход

Фронтенд должен разрабатываться сразу под новый UI и реальный backend (бэкенд), без опоры на моки как основной источник истины.

### 29.2. Требование к backend API

Backend должен уметь явно выдавать данные для следующих представлений:
- overview summary
- task list
- task detail
- approvals list
- artifacts list and detail
- memory list and edit actions
- activity feed
- system health summary
- settings read/update

### 29.3. Отдельный документ

Изменения backend API, необходимые для нового фронта, должны быть оформлены отдельным файлом. Этот документ только фиксирует, что такие изменения ожидаются и проект должен учитывать их с самого начала.

## 30. Definition of Done (критерии готовности)

Фича считается реализованной корректно, если:
- уложена в FSD-структуру
- использует только разрешённые UI-примитивы
- не содержит inline styles и произвольных цветов
- использует токены и общую тему
- не нарушает Telegram-first модель
- покрыта обязательными Playwright tests
- имеет предсказуемые loading / empty / error states
- не скрывает критичную информацию о статусе, результате и необходимости пользовательского действия

---

## 31. Инструкции для AI-кодера

### 31.1. Всегда делать

- начинать с доменной модели экрана
- использовать FSD-слои осознанно
- использовать `shared/ui` и токены
- проектировать страницу как состояние и контент, а не как произвольную верстку
- явно реализовывать loading / empty / error states
- добавлять `data-testid` для критичных элементов

### 31.2. Никогда не делать

- добавлять создание задач в web UI
- добавлять поле отправки сообщения агенту
- превращать task detail в чат
- использовать произвольные стили
- импортировать PrimeVue напрямую вне `shared/ui`
- смешивать технологический шум с основным пользовательским слоем

### 31.3. При сомнении

Если выбор неочевиден, нужно предпочитать решение, которое:
- усиливает прозрачность
- усиливает контроль пользователя
- делает результат понятнее
- не создаёт ложного ощущения активного управления агентом через веб

## 32. Итоговая формулировка

Butler Web UI — это не интерфейс общения с агентом, а интерфейс наблюдения, контроля, подтверждения, анализа результатов и управления памятью.

Пользователь взаимодействует с агентом через Telegram. Через web UI он получает:
- прозрачность исполнения
- контроль чувствительных действий
- доступ к результатам
- анализ ошибок
- контроль памяти
- понимание состояния системы

Именно эта модель должна лежать в основе всех архитектурных, визуальных и инженерных решений фронтенда.
